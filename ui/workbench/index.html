Guide

How to use this workbench

The Workbench is a calm, local-first shell over the local Spectra daemon. It shows where a work session stands and what is safe to do next — it never changes your repos, and the only writes are explicit, audited local-attachment actions (import, rename, tags). Navigate with the left rail or press 1–8. Everything follows one rule: review first — you see a preview and approve before anything runs.

Session History
Where you are now: workspace snapshot, recent checkpoints, conversations, and attachments, plus the next smallest safe action.
Conversations
Project memory threads — each message and the context attached to it. Read-only.
Attachments
Local project artifacts with metadata and an audit trail. Adding a file is explicit: preview, then Import. No delete, move, or remote providers.
Approvals
Actions waiting on your yes/no. Each shows its risk tier, affected files, and a preview before it can run.
Changes
A live timeline of provenance events — what happened, when, and which capability or checkpoint it relates to. Updates without a reload.
Capabilities
What Spectra can do, generated from capability manifests: load mode, approval class, and local/remote boundary.
Settings
Workbench and safety toggles (low-power, preview-only, music-safe, and so on). Placeholders until future daemon state controls them.

Reading an approval

Risk tiers & the review gate

Each approval card carries a coloured risk tier on its left edge and a preview of the proposed work. Nothing executes until you approve it, and every action has an equivalent prism CLI command — so the workbench is never a black box.

green
Low risk — local, observe or preview only. Safe to approve routinely.
amber
Medium risk — local writes or larger jobs. Read the preview before approving.
red
High risk — destructive, remote, or expensive. Approve only deliberately; these stay gated by design.

Live preview & the change pipeline

Local dev only
Launch previews withAI_FORGE_APP_PREVIEW=1and the Workbench watcher withAI_FORGE_WORKBENCH_WATCH=1.

When a preview is running and a validation command is configured, saving a watched file runs that check first — you'll see change → validating → passed/failed in the Changes timeline, and a failing check holds the reload so the last good preview stays up. With no command configured, the preview just reloads as before. Commands come only from your local, git-ignored preview config — never from a request, repo, or AI.
